Before You Write a Single Line of Code, Get Your Environments Right

Most Power Platform projects fail at deployment — not because of bad code, but because of decisions made in the first five minutes that nobody thought to question.

Ahmed El-Khatib
March 2026
12 min read

I've walked into at least a dozen projects where the first thing I see is org32737.crm4.dynamics.com in the browser tab of a production environment. No security groups. Everything built in the default environment. And a Dynamics 365 toggle that someone skipped over during setup — which means you can't install any Dynamics apps now, no matter how much you need them.

These aren't edge cases. They're the norm. And they're all decisions made in the first five minutes of a project — when nobody was paying attention, when the goal was just to "get started."

This post is about those decisions. Not the code. Not the pipelines. The stuff that happens before any of that — and that you can never fully undo once it's done.

What this post covers

Environment types and when to use each · The four most painful environment setup mistakes · A complete environment topology for enterprise projects · A pre-start checklist you can use right now

§ 01

Why Environments Are an Architecture Decision

Most developers treat environments like parking spaces — you just grab one and start. But in Power Platform, every environment is a deployment boundary, a security boundary, and a licensing boundary all at once.

The choices you make when creating an environment are largely irreversible. You can't move a production Dataverse database to a different environment. You can't add Dynamics 365 apps to an environment that wasn't created with that option enabled. And you can't rename the organization URL — not without Microsoft support, not without downtime, and sometimes not at all.

So let's talk about what these decisions actually are.

The Environment Types You Need to Know

Type Dataverse D365 Apps Purpose
Default ✅ Always ✗ Personal maker experiments only. Never for team projects.
Developer ✅ Always Optional Personal dev sandbox. One per maker. Building and previewing is free — running assets requires premium licensing. Never use as a shared team environment.
Sandbox Optional Optional The correct choice for shared team testing. Can be reset — which wipes everything, including data your UAT users are actively entering. Should mirror production config.
Production Optional Optional Live workloads only. Cannot be reset. Full backup and SLA. Note: for any serious solution, Dataverse is effectively required — "optional" only applies to simple automation-only workloads.
Trial ✅ Always Optional 30-day POCs. Never for real projects. Expires automatically.
A note on environment types for ALM

Your Dev, Test, UAT, and integration environments should all be provisioned as production-type environments — not Sandbox. Sandbox environments can be reset and have no SLA. Use production-type environments for any environment that holds team work or is part of your deployment pipeline. Sandbox type is fine for short-lived spike or experiment environments.

Managed Environments — What It Is and Why It Matters

Managed Environments is a premium governance capability that unlocks features you need for any serious project: solution checker enforcement as a deployment gate, sharing limits on canvas apps, maker welcome content, environment-level analytics, and — critically — the ability to use Power Platform Pipelines.

As of early 2026, Pipelines require all target environments to be Managed Environments. This isn't a soft recommendation — if your Test, UAT, or Production environments aren't managed, you can't use pipelines to deploy to them. Plan for this upfront.

What Managed Environments does NOT automatically do: it doesn't turn itself on across all your environments. You enable it per environment, in the Power Platform Admin Center, under each environment's settings. The cost is covered by your existing Power Apps/Power Automate premium licenses for users running assets in those environments.

If you're using Environment Groups, all environments assigned to a group inherit the group's Managed Environments rules automatically. This is the cleanest way to ensure consistent governance without configuring each environment by hand.

For projects that integrate with external systems — Dynamics 365 F&O, third-party ERPs, legacy APIs, Azure services — you'll also want a dedicated Integration / Staging environment between Dev and Test. This gives you a stable, isolated target for integration testing that doesn't pollute your main test environment with data sync noise, incomplete reference data, or webhook misfires. F&O is the most common case, but it's not the only one. If your project has any external system dependency that runs its own test environment, plan for an Integration environment.

Environment Groups

One more thing to plan before you create anything: how your environments will be governed together. Environment Groups — now generally available — let you group related environments and apply a consistent set of governance rules across all of them in one place: sharing limits, solution checker enforcement, AI-generated summaries, and more. Every environment in the group inherits the same baseline configuration automatically, with no manual setup per environment.

For a typical project, create one group covering your Test, UAT, and Production environments. Your Dev environments can live in a separate group with relaxed rules, or be individually managed. Plan the group structure before you create the environments — moving environments between groups later is possible, but retrofitting governance to environments that have already been in active use is painful.

§ 03

The Four Mistakes That Will Haunt You

Every one of these I've seen on real projects. I'm documenting them here so you don't have to learn them the same way.

Mistake 01
Building in the Default Environment

The default environment is always there, always accessible, and always a trap. It's shared across your entire tenant. Everyone with a Power Platform license can see it. There are no security groups, no deployment boundaries, and no clean way to move what you build out of it later.

I've seen teams build entire applications in the default environment because it was "just a proof of concept" — and then the client loved it and wanted it in production. The migration is painful. The cleanup is worse. Budget two to three weeks you didn't plan for, on work that produces nothing new.

✓ The fix

Create a dedicated environment for every project from day one. Reserve the default environment for individual, personal experimentation only, and enforce what can and cannot be built there with a DLP policy.

If you're managing a team or a whole tenant — not just a single project — there's now a better answer: Environment Routing. This is a tenant-level setting that automatically redirects any maker who visits the Power Apps or Power Automate portal into their own personal developer environment, instead of the default. The default environment stops being a dumping ground by default, without relying on everyone reading a governance policy and doing the right thing.

Environment Routing is generally available and is enabled in the Power Platform Admin Center under Tenant Settings. You can scope it to all makers or to specific security groups. Newly created developer environments through routing are Managed Environments by default and can be automatically assigned to an Environment Group. For teams with more than a handful of makers, this is the right answer to Mistake 01 — not just a policy reminder.

One important note: developer environments created through routing require premium licensing to run assets, even without premium connectors. Make sure your licensing plan accounts for this before you enable it broadly.

Mistake 02
No Security Group on the Environment

When you create an environment without assigning a security group, every licensed user in your tenant can access it. This isn't a hypothetical risk — it means developers can stumble into a UAT environment, users can access a test environment with incomplete configurations, and there's no clear boundary between who belongs where.

Fig. 2 — The pulsing red zone illustrates that without a security group, the environment boundary does not exist.

The fix — 5 minutes upfront, hours saved at go-live

Before you create the environment, create the security groups in Entra ID. Name them consistently — something like PP-[Project]-[Env]-[Role]. Assign them at environment creation time. If you skip this and need to add a security group later, you can — but every user currently in the environment without a license or role assignment gets removed, which means an unplanned access-restoration exercise mid-project.

Mistake 03
Skipping the "Enable Dynamics 365 Apps" Toggle

This is the irreversible one. When creating a Dataverse environment, there's a toggle to enable Dynamics 365 apps. Most people skip it because they're "just building a Power App." Then six months later, the client wants to add Dynamics 365 Sales, or integrate with F&O, or install a Dynamics-dependent solution.

You cannot enable Dynamics 365 apps on an existing environment that was created without this option. You have to create a new environment, migrate everything, and update every integration. I've seen projects delayed by weeks because of this single skipped toggle.

What I saw on a real project

A client was six months into a Power Apps build when the business decided to add Dynamics 365 Sales. The environment had been created without the toggle enabled. We spent two weeks planning and executing a migration — exporting solutions, recreating the environment from scratch, re-importing, updating every integration endpoint, reassigning security roles. Two weeks of engineering time, no new functionality delivered, and a go-live pushed back a month. All for a checkbox that takes three seconds to enable at creation time.

Enable it if there is any chance — any chance at all — that Dynamics 365 apps will be needed. The cost at creation time is zero. The cost of not enabling it is measured in weeks.

Mistake 04
Not Setting the Organization URL

When you create an environment, Power Platform auto-generates an organization URL — something like org32737.crm4.dynamics.com. Nobody changes it because nobody notices it. And then it ends up in production, in API calls, in documentation, in client-facing URLs, and in the browser tabs of every user on the project.

Changing it later requires a Microsoft support request, planned downtime, and updates to every integration that references the URL. On some configurations it can't be changed at all without recreating the environment.

✗ Auto-generated (don't use)
org32737.crm4.dynamics.com
✓ Set at creation (always do this)
contoso-prod.crm4.dynamics.com
✓ The fix

Always set the URL manually at creation time. Use a consistent naming convention: [client]-[env] — e.g. contoso-dev, contoso-uat, contoso-prod. Two minutes of work, permanent benefit.

§ 03.5

The Fifth Mistake Nobody Talks About: No DLP Policy

I said four mistakes. There's a fifth one I left off the numbered list because it doesn't happen at environment creation time — it happens in the two hours after. And by then, makers are already inside the environment, building things, connecting things, and moving data in directions you haven't thought about yet.

Data Loss Prevention policies are as foundational as the environment itself. A DLP policy defines which connectors are available in an environment, which ones can work together, and which ones are blocked entirely. No DLP policy means makers can build flows that send your Dataverse customer records to a personal Gmail account, pull data into a public-facing HTTP endpoint, or connect internal systems to consumer services that have no place near your data.

This isn't hypothetical. It's the most common governance gap I see on projects that got the environments right but skipped the policies.

How DLP policies work

Every connector gets assigned to one of three buckets: Business, Non-Business, or Blocked. The rule is simple: connectors from different buckets can't be used together in the same app or flow. You configure policies at two levels:

Tenant-level policy — applies to all environments in your tenant. Use this for universal high-risk blocks (personal email, social media, consumer storage). This is your wall.

Environment-level policy — applies to a specific environment only. Use this to restrict further or to allow specific connectors for a project's needs.

Environment-level policies can only make things more restrictive than the tenant policy — they cannot grant permissions the tenant policy has blocked.

What to do before you hand the environment to a maker

Define your DLP policy before a single maker has access to the environment. For most project environments the starting position is: put all Microsoft/Dataverse connectors in Business, move all third-party and consumer connectors to Blocked, then work with your integration requirements to move specific connectors to Business as needed. Lock the default group to Blocked so any new connector added by Microsoft lands in the restricted bucket until you explicitly approve it.

The default environment needs its own strict DLP policy — the most restrictive one in your tenant. It's accessible to everyone, has no security group, and is exactly where an untrained maker will go if Environment Routing isn't enabled.

DLP is a continuous process, not a setup-and-forget step. New connectors are added to the platform regularly. Make reviewing your DLP policies part of your project governance cadence.

The craft principle

A well-configured environment takes a few hours of deliberate work across a few people — architect, Entra admin, Power Platform admin. The decisions made in that window are largely permanent. A poorly-set environment doesn't just cost you time at go-live. It costs you credibility, client trust, and sometimes the project. This is where quality starts.
